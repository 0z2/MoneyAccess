# Разработка T Design System

## Запуск dev сервера

```bash
npm install
npm run dev
```

Откроется preview-приложение с вкладками:
- `Стили` — токены и базовые style-артефакты
- `Атомы` — базовые UI-элементы
- `Компоненты` — более сложные компоненты и формы

## Архитектура проекта

```
src/
├── assets/
│   ├── Style/          # Глобальные токены: цвета, шрифты, радиусы, тени, размеры
│   ├── Icon/           # Общие иконки в icons.tsx и контейнер icon.css
│   └── Font/           # Файлы шрифтов
├── components/         # Каждый компонент в своей папке
└── index.ts            # Единая точка экспорта компонентов

demo/
├── preview/
│   ├── StylesPreview.tsx
│   ├── AtomsPreview.tsx
│   └── ComponentsPreview.tsx
├── App.tsx             # Общий каталог-превью с вкладками
└── index.css           # Подключение токенов и компонентов для preview

README.md               # Верхнеуровневое описание UI Kit
```

Собранный CSS библиотеки подключается потребителем через `@PluginWoman/t-ds/style`. В исходниках все глобальные токены лежат в `src/assets/Style/`, компоненты — в `src/components/<Name>/`.

## Добавление нового компонента

1. **Сними компонент из Figma:**
   - используй `get_design_context`
   - используй `get_screenshot`
   - используй `get_metadata`
   - если `get_design_context` вернул слишком большой или шумный результат, сначала найди нужные дочерние узлы через `get_metadata`, а потом сними их отдельно
   - screenshot из Figma держи как визуальный source of truth до конца реализации, а не только как стартовую подсказку

### Как работать с макетом в Figma

- Всегда начинай с конкретного component/frame node, а не со всей страницы или соседних секций
- Если пользователь дал assembled example, сначала сними собранный узел, затем отдельно сними внутренние части: header, content, footer, button, icon
- Сначала считывай структуру и поведение, потом стили:
  - какие части компонента обязательны, а какие опциональны
  - какие варианты реально управляются prop-ами
  - где компонент составной и должен собираться из подкомпонентов
  - какие состояния есть: default, hover, disabled, loading, danger и т.д.
- Не копируй MCP-код из Figma как финальную реализацию 1:1. Он нужен как референс структуры, размеров и токенов, но итоговый код должен быть переписан в стиль этого репозитория
- Не тащи в код временные asset URL, абсолютные размеры всего экрана, tailwind-классы или случайные figma-имена слоёв
- Всегда проверяй, не является ли узел только примером наполнения, а не API компонента. Например:
  - текст из примера не всегда должен становиться дефолтным значением
  - количество кнопок в assembled example не всегда означает фиксированную структуру
  - наличие иконки в примере не означает, что иконка всегда обязательна
- Если в Figma есть pattern или usage description, опирайся на него при проектировании API компонента, а не только на внешний вид
- Если в Figma есть variant/property, сначала проверь, нужно ли это реально выводить в публичный prop:
  - если это пользовательски управляемая часть компонента, делай prop
  - если это только внутреннее состояние или часть фиксированного UI, не расширяй API без необходимости
- Если макет конфликтует с токенами библиотеки, сначала ищи уже существующий токен или паттерн в `src/assets/Style/` и соседних компонентах, а не добавляй локальные захардкоженные значения
- Если точный размер или отступ виден в макете, сначала проверь, есть ли ему прямое соответствие в токенах `spacing`, `radius`, `font`, `color`
- Для составных компонентов обязательно проверяй:
  - как выглядит assembled example
  - как выглядят отдельные части
  - как компонент должен вести себя при отключении опциональных блоков
  - не ломается ли ритм отступов и высота секций в этих состояниях
- После реализации сравни компонент с Figma не только в дефолтном состоянии, но и в тех состояниях, которые ты добавил в API и preview

2. **Создай папку компонента:**
   ```
   src/components/MyComponent/
   ├── MyComponent.tsx
   ├── my-component.css
   └── README.md
   ```

3. **Реализуй компонент в `.tsx`:**
   ```tsx
   import React from 'react';
   import './my-component.css';

   interface MyComponentProps {
     variant?: 'primary' | 'secondary';
     children: React.ReactNode;
   }

   export const MyComponent: React.FC<MyComponentProps> = ({
     variant = 'primary',
     children
   }) => {
     return <div className={`my-component my-component--${variant}`}>{children}</div>;
   };
   ```

### Правила именования props

- Булевый prop, который управляет показом части компонента, называй через `hasName`: `hasDescription`, `hasIcon`, `hasRightAccessory`, `hasChevron`.
- Булевый prop, который управляет состоянием или переключает вид компонента, называй через `isName`: `isOpen`, `isSelected`, `isDisabled`, `isLoading`, `isError`, `isHorizontal`, `isAdaptive`.
- Не используй `show*`, `hide*`, голые `checked`, `disabled`, `open`, `selected` в публичном API компонента. Используй `has*` или `is*`.
- Если prop выбирает один вариант из нескольких, используй `variant`: `variant="primary"`, `variant="danger"`, `variant="steps"`.
- Если prop выбирает структуру или раскладку компонента, используй `layout`: `layout="1-button"`, `layout="2-horizontal-buttons"`.
- `type` допустим только для нативного HTML-смысла, например `Button` прокидывает `type="button" | "submit" | "reset"` в `<button>`.
- Для вложенных action/item-объектов сохраняй те же правила: `isDisabled`, `isLoading`, `isSelected`, `hasBadge`.
- Для uncontrolled-состояний можно использовать React-паттерны `defaultOpen`, `defaultValue`, `defaultSelected`.
- Если название из Figma не совпадает с правилами API, переименуй его в коде и README по этим правилам.

4. **Используй только существующие дизайн-токены из `src/assets/Style/`:**
   ```css
   .my-component {
     color: var(--primitive-primary);
     padding: var(--spacing-4x);
     border-radius: var(--rounding-6x);
   }
   ```

5. **Экспортируй компонент в `src/index.ts`:**
   ```typescript
   export { MyComponent } from './components/MyComponent/MyComponent';
   ```

6. **Добавь preview в правильную вкладку:**
   - если добавляешь или меняешь токены, типографику, цвета, радиусы, тени или размеры — `demo/preview/StylesPreview.tsx`
   - если компонент является базовым атомом — `demo/preview/AtomsPreview.tsx`
   - если компонент составной, сценарный или относится к формам/сложным блокам — `demo/preview/ComponentsPreview.tsx`

   Если пользователь явно не сказал, куда расположить preview, сначала уточни у него нужную вкладку: `Стили`, `Атомы` или `Компоненты`.

   В demo все интерактивные элементы должны работать как в реальном интерфейсе: клики, переключения, ввод, открытие списков и другие состояния не имитируются статичной разметкой, а поддерживают живое поведение.

   Если задача касается только локальной разработки или личного preview, изменения в `demo/` держим локально и не переносим в библиотечную сборку.

7. **Если компонент интерактивный, добавь поведение рядом с компонентом или в общий слой только при повторяемом паттерне.**

8. **Добавь или обнови `README.md` внутри папки компонента.**

9. **Если компонент использует иконки, проверь правила из раздела `Иконки`:**
   - повторяемые иконки добавляй в `src/assets/Icon/icons.tsx`
   - экспортируй их из `src/index.ts`
   - в компонент передавай иконку через заменяемый `ReactNode`, если дизайн компонента действительно предусматривает иконку
   - не добавляй icon API в компоненты, где по дизайну иконки быть не должно, например в `Button`

## Правила по стилям

- Шрифты брать из `src/assets/Style/font.css`
- Цвета брать из `src/assets/Style/color.css`
- Радиусы брать из `src/assets/Style/radius.css`
- Размеры и отступы брать из `src/assets/Style/spacing.css`
- Тени брать из `src/assets/Style/shadow.css`
- Не заводить локальные дубли глобальных токенов
- Не создавать глобальные стили внутри компонентов
- Использовать только локальные классы: `.component-name`, `.component-name__element`, `.component-name--modifier`
- Hover-состояния обычно не показывать в demo как отдельный пример, но сам `:hover` в CSS сохранять
- Локальные `:hover`, `:focus`, `:active` писать только если это уникальное состояние конкретного компонента
- Для macOS рендер текста уже настроен на уровне preview-приложения, не дублируй его в компонентах

## Правила по общим interactive-стилям

- Все повторяющиеся `hover`, `focus`, `active` эффекты сначала искать среди уже существующих общих utility-классов
- Если для паттерна уже есть общий стиль, обязательно переиспользуй его, а не дублируй локально в компоненте
- Для общих hover-эффектов используй единый utility-класс
- Если появляется новый повторяемый interactive-паттерн, сначала вынеси его в общую утилиту, и только потом применяй в компонентах

Текущие общие utility-паттерны:
- `hoverOpacity` — общий hover-эффект снижения непрозрачности, использовать для интерактивных иконок и похожих легких hover-сценариев

## Иконки

Контейнер с иконкой — технический компонент для масштабирования иконки до нужного размера. Контейнер может быть 5 размеров: `L (32px)`, `M (24px)`, `S (20px)`, `XS (16px)`, `2XS (12px)`. Размеры контейнера соответствуют размерам существующих наборов иконок.

В контейнер любого размера можно поместить иконку любого размера. Если иконка меньше контейнера, она растягивается до размера контейнера. Если больше — сжимается. Это работает за счет `.ds-icon`, который задает размер контейнера, и правила `.ds-icon svg { width: 100%; height: 100%; }`.

Если иконка визуально не по центру, сначала проверь `viewBox` и координаты самого SVG. `.ds-icon` центрирует SVG-элемент, но не исправляет смещенную геометрию внутри `viewBox`.

Часто повторяющиеся иконки нужно выносить в `src/assets/Icon/icons.tsx` и экспортировать из `src/index.ts`. В компонентах и demo такие иконки подключаются как готовые JSX-компоненты, чтобы не дублировать SVG-код.

Если иконка приходит из Figma:
- Выдели именно icon node, а не весь компонент или frame
- Скопируй ее как SVG: `Copy as SVG`
- Перенеси SVG в JSX, сохрани `viewBox`
- Если Figma показывает `Icon Container`, используй его размерную сетку: `L (32)` → `.ds-icon--l`, `M (24)` → `.ds-icon--m`, `S (20)` → `.ds-icon--s`, `XS (16)` → `.ds-icon--xs`, `2XS (12)` → `.ds-icon--2xs`
- Не используй временные `<img src="https://www.figma.com/api/mcp/asset/...">` из сгенерированного MCP-кода как финальную реализацию иконки
- Убери захардкоженные `fill`, `stroke`, `width`, `height`, если они мешают наследованию размера и цвета
- Замени цветовые значения на `currentColor`: `fill="currentColor"` или `stroke="currentColor"`
- Если SVG состоит из `path`, обычно достаточно не указывать `fill`: общий `.ds-icon svg` уже задает `fill: currentColor`
- Передавай иконку в компонент как JSX через prop, а не как путь к файлу и не как `<img>`
- Если компонент по дизайну имеет пользовательски заменяемую иконку или аксессуар, он должен принимать заменяемый `ReactNode`, например `icon`, `leftIcon`, `left`, `right`
- Если иконка является внутренним фиксированным состоянием компонента, например `CheckmarkIcon` и `MinusIcon` в `Checkbox`, ее можно подключать внутри компонента из `src/assets/Icon/icons.tsx`
- Если по дизайну компонент не поддерживает иконку, не добавляй для нее отдельный prop. Например, `Button` не принимает иконку

Повторяемые иконки подключай из `src/assets/Icon/icons.tsx` и оборачивай в `.ds-icon` из `src/assets/Icon/icon.css`:

```tsx
import { CircleIcon } from '../../assets/Icon/icons';

<span className="ds-icon ds-icon--m" aria-hidden="true">
  <CircleIcon />
</span>
```

Inline SVG допустим как исходник для новой иконки в `icons.tsx` или для редкого уникального случая, который не повторяется.

SVG должны краситься через `currentColor`.

## Публикация новой версии

Не публикуй пакет, не создавай git tag и не запускай релиз автоматически после добавления компонента или обновления preview.

Публикацию можно делать только после непосредственной команды пользователя на публикацию или релиз. До этого допустимы только локальные проверки: `npm run type-check`, `npm run build`, запуск preview и ревью изменений.

Когда пользователь прямо попросил опубликовать:

1. Обнови версию в `package.json`
2. Создай тег на GitHub
3. Запушь тег
4. Дождись результата GitHub Actions

GitHub Actions автоматически собирает библиотеку и публикует ее на GitHub Packages.
